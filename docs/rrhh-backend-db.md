# RRHH Backend y Base de Datos

Este documento describe los cambios necesarios para que el modulo de RRHH funcione como una nomina real con historial laboral, contratos por periodo y datos consistentes para calculos posteriores.

## Reglas Principales

- `EMP_TIPO_CONTRATO` es catalogo. No debe relacionarse directamente con un empleado.
- `EMP_EMPLEADO_CONTRATO` es la fuente real del historial de contratos.
- Cada empleado debe tener como maximo un contrato vigente con `TCO_ES_ACTUAL = 1`.
- Al cambiar contrato, el backend debe cerrar el periodo anterior y crear uno nuevo en una transaccion.
- No se deben eliminar fisicamente registros historicos en produccion; se deben anular o inactivar.
- Nomina debe calcularse contra datos vigentes dentro del periodo de nomina, no solo contra el estado actual del empleado.
- Los ingresos y egresos variables deben asignarse primero a empleado/periodo. `EMP_NOMINA_DETALLE` debe ser resultado final de la generacion, no la pantalla principal de captura operativa.

## Flujo Recomendado de Nomina

1. Crear catalogos de ingresos y descuentos:
   - `EMP_INGRESO`
   - `EMP_DESCUENTO`
2. Asignar ingresos/descuentos variables por empleado y periodo.
3. Generar nomina del periodo.
4. El backend crea `EMP_NOMINA` y `EMP_NOMINA_DETALLE` desde:
   - contrato vigente;
   - salario base;
   - ingresos/descuentos asignados;
   - descuentos obligatorios;
   - KPI;
   - prestamos;
   - asistencia/suspensiones.
5. Contabilidad revisa la planilla en estado `B` (Borrador).
6. Contabilidad envia a gerente: estado `P` (Pendiente).
7. Gerencia aprueba/rechaza: estado `A` o `I`.
8. Solo con estado `A` y detalle cuadrado se descarga CSV.

## Flujo Cambio de Contrato

Endpoint recomendado:

```http
PUT /empleados/:id
```

Cuando el payload incluya cambio en `tic_id`, `emp_fecha_inicio_contrato` o `emp_fecha_fin_contrato`:

1. Iniciar transaccion.
2. Buscar contrato actual:
   ```sql
   SELECT *
   FROM EMP_EMPLEADO_CONTRATO
   WHERE EMP_ID = :emp_id
     AND TCO_ES_ACTUAL = 1;
   ```
3. Si existe contrato actual y cambio el contrato:
   - cerrar contrato anterior con `TCO_FECHA_FIN = fecha_inicio_nuevo - 1 dia`;
   - marcar `TCO_ES_ACTUAL = 0`;
   - guardar motivo si aplica.
4. Crear nuevo registro en `EMP_EMPLEADO_CONTRATO`.
5. Actualizar `EMP_EMPLEADO` con el contrato vigente.
6. Confirmar transaccion.

Payload esperado desde frontend:

```json
{
  "tic_id": "2",
  "emp_fecha_inicio_contrato": "2026-05-12",
  "emp_fecha_fin_contrato": "",
  "emp_motivo_cambio_contrato": "Cambio a contrato indefinido"
}
```

## Tablas Recomendadas

### EMP_EMPLEADO

```sql
CREATE TABLE EMP_EMPLEADO (
  EMP_ID NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  EMP_NOMBRE VARCHAR2(100) NOT NULL,
  EMP_APELLIDO VARCHAR2(100) NOT NULL,
  EMP_DPI VARCHAR2(20) NOT NULL UNIQUE,
  EMP_NIT VARCHAR2(20),
  EMP_TELEFONO VARCHAR2(20),
  EMP_FECHA_CONTRATACION DATE NOT NULL,
  EMP_ESTADO CHAR(1) DEFAULT 'A' NOT NULL,
  TIC_ID NUMBER,
  HOR_ID NUMBER,
  PUE_ID NUMBER,
  SED_ID NUMBER,
  DEP_ID NUMBER,
  EMP_SUELDO NUMBER(12,2),
  EMP_FECHA_INICIO_CONTRATO DATE,
  EMP_FECHA_FIN_CONTRATO DATE,
  EMP_FOTO CLOB,
  EMP_FECHA_CREACION DATE DEFAULT SYSDATE,
  EMP_FECHA_MODIFICACION DATE,
  CONSTRAINT FK_EMP_TIC FOREIGN KEY (TIC_ID) REFERENCES EMP_TIPO_CONTRATO(TIC_ID),
  CONSTRAINT FK_EMP_HOR FOREIGN KEY (HOR_ID) REFERENCES EMP_HORARIO(HOR_ID),
  CONSTRAINT FK_EMP_PUE FOREIGN KEY (PUE_ID) REFERENCES EMP_PUESTO(PUE_ID),
  CONSTRAINT FK_EMP_SED FOREIGN KEY (SED_ID) REFERENCES EMP_SEDE(SED_ID),
  CONSTRAINT FK_EMP_DEP FOREIGN KEY (DEP_ID) REFERENCES EMP_DEPARTAMENTO(DEP_ID),
  CONSTRAINT CK_EMP_ESTADO CHECK (EMP_ESTADO IN ('A', 'I'))
);
```

### EMP_TIPO_CONTRATO

```sql
CREATE TABLE EMP_TIPO_CONTRATO (
  TIC_ID NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  TIC_NOMBRE VARCHAR2(100) NOT NULL,
  TIC_NUMERO VARCHAR2(50) NOT NULL,
  TIC_DESCRIPCION VARCHAR2(255),
  TIC_TIPO_JORNADA VARCHAR2(100) NOT NULL,
  TIC_FECHA_MODIFICACION DATE,
  TIC_ESTADO CHAR(1) DEFAULT 'A',
  CONSTRAINT CK_TIC_ESTADO CHECK (TIC_ESTADO IN ('A', 'I'))
);
```

Nota: quitar `EMP_ID` de esta tabla. La relacion empleado-contrato pertenece a `EMP_EMPLEADO_CONTRATO`.

### EMP_EMPLEADO_CONTRATO

```sql
CREATE TABLE EMP_EMPLEADO_CONTRATO (
  TCO_ID NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  EMP_ID NUMBER NOT NULL,
  TIC_ID NUMBER NOT NULL,
  TCO_FECHA_INICIO DATE NOT NULL,
  TCO_FECHA_FIN DATE,
  TCO_ESTADO CHAR(1) DEFAULT 'A' NOT NULL,
  TCO_ES_ACTUAL NUMBER(1) DEFAULT 0 NOT NULL,
  TCO_MOTIVO_CAMBIO VARCHAR2(255),
  TIC_FECHA_MODIFICACION DATE,
  TCO_FECHA_CREACION DATE DEFAULT SYSDATE,
  TCO_FECHA_ACTUALIZACION DATE,
  CONSTRAINT FK_TCO_EMP FOREIGN KEY (EMP_ID) REFERENCES EMP_EMPLEADO(EMP_ID),
  CONSTRAINT FK_TCO_TIC FOREIGN KEY (TIC_ID) REFERENCES EMP_TIPO_CONTRATO(TIC_ID),
  CONSTRAINT CK_TCO_ESTADO CHECK (TCO_ESTADO IN ('A', 'I')),
  CONSTRAINT CK_TCO_ACTUAL CHECK (TCO_ES_ACTUAL IN (0, 1)),
  CONSTRAINT CK_TCO_FECHAS CHECK (TCO_FECHA_FIN IS NULL OR TCO_FECHA_FIN >= TCO_FECHA_INICIO)
);
```

Indice unico recomendado para un solo contrato actual:

```sql
CREATE UNIQUE INDEX UX_TCO_EMPLEADO_ACTUAL
ON EMP_EMPLEADO_CONTRATO (
  CASE WHEN TCO_ES_ACTUAL = 1 THEN EMP_ID END
);
```

### EMP_KPI_RESULTADO

```sql
ALTER TABLE EMP_KPI_RESULTADO ADD EMP_ID NUMBER;
ALTER TABLE EMP_KPI_RESULTADO
  ADD CONSTRAINT FK_KRE_EMP FOREIGN KEY (EMP_ID) REFERENCES EMP_EMPLEADO(EMP_ID);
```

El frontend de resultados KPI ya trabaja con empleado, por eso `EMP_ID` debe existir.

### EMP_CUENTA_BANCARIA

```sql
ALTER TABLE EMP_CUENTA_BANCARIA ADD CUE_ES_PRINCIPAL NUMBER(1) DEFAULT 1;
ALTER TABLE EMP_CUENTA_BANCARIA ADD CUE_ESTADO CHAR(1) DEFAULT 'A';
ALTER TABLE EMP_CUENTA_BANCARIA
  ADD CONSTRAINT CK_CUE_ESTADO CHECK (CUE_ESTADO IN ('A', 'I'));
```

Recomendacion: no guardar `CUE_ID` en `EMP_EMPLEADO`. La cuenta se relaciona desde `EMP_CUENTA_BANCARIA.EMP_ID`.

### EMP_NOMINA_ASIGNACION

Tabla recomendada para capturar ingresos y egresos variables antes de generar la nomina. Esto evita editar directamente el detalle final.

```sql
CREATE TABLE EMP_NOMINA_ASIGNACION (
  NAS_ID NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  PER_ID NUMBER NOT NULL,
  EMP_ID NUMBER NOT NULL,
  TIS_ID NUMBER,
  TDS_ID NUMBER,
  NAS_TIPO CHAR(1) NOT NULL,
  NAS_MONTO NUMBER(12,2) NOT NULL,
  NAS_CANTIDAD NUMBER(12,2),
  NAS_REFERENCIA VARCHAR2(100),
  NAS_DESCRIPCION VARCHAR2(255),
  NAS_ESTADO CHAR(1) DEFAULT 'A' NOT NULL,
  NAS_FECHA_CREACION DATE DEFAULT SYSDATE,
  NAS_FECHA_ACTUALIZACION DATE,
  CONSTRAINT FK_NAS_PER FOREIGN KEY (PER_ID) REFERENCES EMP_PERIODO(PER_ID),
  CONSTRAINT FK_NAS_EMP FOREIGN KEY (EMP_ID) REFERENCES EMP_EMPLEADO(EMP_ID),
  CONSTRAINT FK_NAS_TIS FOREIGN KEY (TIS_ID) REFERENCES EMP_INGRESO(TIS_ID),
  CONSTRAINT FK_NAS_TDS FOREIGN KEY (TDS_ID) REFERENCES EMP_DESCUENTO(TDS_ID),
  CONSTRAINT CK_NAS_TIPO CHECK (NAS_TIPO IN ('I', 'D')),
  CONSTRAINT CK_NAS_ESTADO CHECK (NAS_ESTADO IN ('A', 'I')),
  CONSTRAINT CK_NAS_CONCEPTO CHECK (
    (NAS_TIPO = 'I' AND TIS_ID IS NOT NULL AND TDS_ID IS NULL)
    OR
    (NAS_TIPO = 'D' AND TDS_ID IS NOT NULL AND TIS_ID IS NULL)
  )
);
```

Indice recomendado para evitar duplicar el mismo concepto activo al empleado en el mismo periodo:

```sql
CREATE UNIQUE INDEX UX_NAS_PER_EMP_CONCEPTO
ON EMP_NOMINA_ASIGNACION (
  PER_ID,
  EMP_ID,
  NAS_TIPO,
  NVL(TIS_ID, -1),
  NVL(TDS_ID, -1),
  CASE WHEN NAS_ESTADO = 'A' THEN 1 END
);
```

## Validaciones Backend Obligatorias

- No permitir periodos de contrato solapados por empleado.
- No permitir mas de un contrato actual por empleado.
- Si el tipo de contrato es indefinido, guardar `TCO_FECHA_FIN` como `NULL`.
- Si el contrato no es indefinido, exigir fecha fin.
- Si se cambia contrato, exigir `emp_motivo_cambio_contrato`.
- No eliminar registros historicos si ya fueron usados en nomina.
- Auditar cambios sensibles: contrato, sueldo, cuenta bancaria, puesto, sede y horario.
